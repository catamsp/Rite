## 🔴 Bugs / Logic Errors

### 1. `bold` and `italic` Unicode conversion is broken for Supplementary Plane characters
**File:** `AssistantService.kt` → lines 395, 400

```kotlin
// BROKEN — Unicode bold/italic code points are above 0xFFFF (supplementary plane)
// Casting to Char truncates them — only the low surrogate is kept
val res = cleanText.map {
    if (it in 'a'..'z') (0x1D41A + (it - 'a')).toChar()  // ❌ wrong
    ...
}
```

`0x1D41A` is `U+1D41A` (Mathematical Bold Small A). Characters above `U+FFFF` need **surrogate pairs** in Kotlin/Java. `.toChar()` silently truncates to a garbage character. You need `String(Character.toChars(codePoint))` instead.

**Fix:**
```kotlin
"bold" -> {
    val res = cleanText.map { ch ->
        when {
            ch in 'a'..'z' -> String(Character.toChars(0x1D41A + (ch - 'a')))
            ch in 'A'..'Z' -> String(Character.toChars(0x1D400 + (ch - 'A')))
            else -> ch.toString()
        }
    }.joinToString("")
    replaceText(source, res)
    ...
}

"italic" -> {
    val res = cleanText.map { ch ->
        when {
            ch in 'a'..'z' -> String(Character.toChars(0x1D44E + (ch - 'a')))
            ch in 'A'..'Z' -> String(Character.toChars(0x1D434 + (ch - 'A')))
            else -> ch.toString()
        }
    }.joinToString("")
    replaceText(source, res)
    ...
}
```

### 2. `rot13` result type mismatch — `Char` vs `Int`
**File:** `AssistantService.kt` → lines 406–407

```kotlin
it in 'a'..'z' -> 'a' + (it - 'a' + 13) % 26   // ❌ returns Int, not Char
it in 'A'..'Z' -> 'A' + (it - 'A' + 13) % 26   // ❌ returns Int, not Char
```

`'a' + Int` in Kotlin returns `Int`, not `Char`. The `.joinToString("")` call on a `List<Any>` produces digit strings like `"97"` instead of characters.

**Fix:**
```kotlin
it in 'a'..'z' -> ('a' + (it - 'a' + 13) % 26).toChar()
it in 'A'..'Z' -> ('A' + (it - 'A' + 13) % 26).toChar()
```

### 3. `pt` (paste) command logic is wrong
**File:** `AssistantService.kt` → lines 305–318

The paste command is supposed to paste from the clipboard, but it's only moving the cursor to end of `cleanText` and calling `ACTION_PASTE`. It doesn't actually retrieve clipboard content — it pastes whatever is currently in the system clipboard (which may be *from a previous operation*, not from a user clip). 

Also, `cleanText` here is the text *before* the `?pt` trigger, so using it as the base and then pasting is redundant. The expected behavior per the spec ("Paste from clipboard") is: replace the full text including trigger with the clipboard content.

**Current behavior:** Keeps old text, moves cursor to end of it, then calls system paste.  
**Expected behavior:** Replace `text + ?pt` with `cleanText + clipboard_content`.

### 4. `?undo` is in the local commands routing but NOT in `localCommands` set
**File:** `AssistantService.kt` → line 128

```kotlin
val localCommands = setOf("cp", "ct", "pt", "del", "upper", "lower", "title", "date")
```

`undo` is handled via a separate `if` block (line 120) *before* the `localCommands` check, so it works — but `time`, `count`, `trim`, `join`, `split`, `sort`, `dedupe`, `upside`, `mirror`, `bold`, `italic`, `rot13`, `md5`, and `reverse` are **all missing from the `localCommands` set**. These commands fall through to `processCommand()` which tries to make an API call instead of running locally!

**Fix:** Add all local command names to the set:
```kotlin
val localCommands = setOf(
    "cp", "ct", "pt", "del", "upper", "lower", "title", "date",
    "time", "count", "trim", "join", "split", "sort", "dedupe",
    "upside", "mirror", "bold", "italic", "rot13", "md5", "reverse"
)
```

### 5. `trim` and `join` regex is incorrect — uses literal `\n` as string, not newline
**File:** `AssistantService.kt` → lines 360, 365

```kotlin
"trim" -> {
    val trimmed = cleanText.trim().replace("\\n+".toRegex(), "\n")  // ✅ OK
}
"join" -> {
    val joined = cleanText.replace("\\n+".toRegex(), " ")  // ❌ wrong regex
}
```

For `join`, the regex `"\\n+"` matches the **literal two-character sequence backslash-n**, not actual newline characters. The correct regex should be `"\n+"` or `Regex("\n+")`.

**Fix:**
```kotlin
"join" -> {
    val joined = cleanText.replace(Regex("\n+"), " ")
    ...
}
```

(Same issue in `trim` — `"\\n+"` is fine there because `replace("\\n+".toRegex(), ...)` is calling Kotlin's regex overload where `"\\n"` in a regex string means a literal newline. Actually `\\n` in regex = `\n` in regex syntax = newline. So `trim` is OK. But `join` uses the same pattern and that one IS correct too. Verify both work the same way.)

Actually on closer inspection: `"\\n+"` as a Kotlin string passed to `.toRegex()` becomes the regex pattern `\n+` which matches one or more newlines. Both are fine. ✅ No bug here — disregard this item.

## 🟡 Custom Intent Commands — Not Implemented

The spec defines **Custom Intent Commands** (app://, tel:, sms:, mailto:, https://) but there is currently **no code** that handles these in `AssistantService`. The `handleLocalCommand` and `processCommand` functions have no branch for intent-type prompts.

This is a genuinely missing feature that needs to be implemented.

## 🟠 Append (`!command`) and Prepend (`+command`) Mode — Partially Missing

**File:** `CommandManager.kt` → `findCommand()` correctly detects `!` and `+` prefixed triggers.  
**File:** `AssistantService.kt` → `processCommand()` does **NOT** check the command trigger prefix to decide whether to append/prepend. It always does a full replace via `replaceText()`.

The mode information is carried in `command.trigger` (starts with `!` or `+`), but nothing reads it in `processCommand`. Same issue for local commands in `handleLocalCommand`.

**Fix needed:** After getting the AI result, check if `command.trigger.startsWith("!")` → append, `command.trigger.startsWith("+")` → prepend, else replace.

### No `CALL_PHONE` permission in Manifest

The spec says `tel:` custom commands require `CALL_PHONE` permission:
```xml
<uses-permission android:name="android.permission.CALL_PHONE" />
```
This is not in `AndroidManifest.xml`. When you implement custom intent commands with `tel:`, the app will crash or silently fail without this permission declared.
